# 小程序面板需求文档标准结构

本文是本技能的唯一结构标准，所有输出都以 Markdown 输出。除非用户明确要求只输出局部章节，否则默认生成完整模板。

## 1. 通用规则

- 使用 Markdown 标题和 Markdown 表格，不使用 HTML table。
- 全文默认顺序固定，不交换章节位置：
  1. 版本记录表
  2. `## 一、需求背景`
  3. `## 二、协作方与交付件`
  4. `## 三、产品总览`
  5. `## 四、小程序功能说明`
  6. `## 五、附件`
- 信息不足时使用明确占位，如 `待业务确认`、`待补充链接`、`待补充示意图`。
- 不编造人名、链接、版本结论、DP 参数或评审结论。

## 2. 标准章节

| 章节 | 是否默认保留 | 内容形式 | 说明 |
| --- | --- | --- | --- |
| 版本记录表 | 是 | 表格 | 全文输出时始终放在最上方 |
| 一、需求背景 | 是 | 段落 | 描述场景、目标用户、需求目的 |
| 二、协作方与交付件 | 是 | 表格 | 标记涉及角色和交付件链接 |
| 三、产品总览 | 是 | 标题 + 表格 | 下含产品基本信息和 DP 点说明 |
| 四、小程序功能说明 | 是 | 分页面标题 + 表格 | 每个页面块一张表 |
| 五、附件 | 是 | 表格 | 无附件也保留占位行 |

## 3. 标准表格与写法

### 3.1 版本记录表

```md
| 版本号 | 时间 | 变更内容 |
| --- | --- | --- |
| V1.0 | YYYY/MM/DD | 初版 |
```

- 用户给出版本信息时直接替换。
- 只输出局部章节时可以省略该表。

### 3.2 二、协作方与交付件

标题下先给简短说明，再输出下表：

```md
| 角色 | 是否涉及 | 交付件 | 备注 |
| --- | --- | --- | --- |
| 产品经理 | √ | 需求文档（本文档） | 待补充 |
| UI | x | [设计稿 / HTML](待补充链接) | 待补充 |
| 小程序 | x | [技术文档](待补充链接) | 待补充 |
| App | x | [技术文档](待补充链接) | 待补充 |
| 云端 | x | [技术文档](待补充链接) | 待补充 |
| 嵌入式 | x | [技术文档](待补充链接) | 待补充 |
| 测试 | x | [测试用例](待补充链接) | 待补充 |
```

- `是否涉及` 使用 `√` 或 `x`。
- 若未知是否涉及，默认保留该行并写 `待业务确认`，不要删除角色。

### 3.3 三、产品总览

`三、产品总览` 下固定保留两个子章节。

#### 3.3.1 `### 1、产品基本信息`

先补一句“以下字段均为必填项”，再输出四列表：

```md
| 字段 | 内容 | 字段 | 内容 |
| --- | --- | --- | --- |
| PID | Wi-Fi PID: 待业务确认；Zigbee PID: 待业务确认 | App 版本要求 | 待业务确认 |
| 通讯能力 | 待业务确认 | 覆盖语言 | 待业务确认 |
| 主题要求 | 待业务确认 | 是否支持群组 | 待业务确认 |
| 新建或迭代 | 待业务确认 | 是否适配 Pad | 待业务确认 |
```

字段口径：

- `PID`：按已知协议拆分填写；未知时保留占位。
- `通讯能力`：写具体协议，如 `Wi-Fi`、`Wi-Fi + Ble`、`Zigbee`、`Matter`。
- `覆盖语言`：写实际语言列表，如 `中文、英文、德语`。
- `主题要求`：常见值如 `浅色模式`、`深色模式`、`跟随 App`。
- `新建或迭代`：常见值如 `全新开发`、`基于已有项目迭代`。

#### 3.3.2 `### 2、DP 点说明`

如果有说明文字，可先写一行“数据传输类型：可下发可上报（rw）/ 只下发（wr）/ 只上报（ro）”，然后输出下表：

```md
| DP ID | 功能点名称 | 标识符 | 数据传输类型 | 数据类型 | 功能点属性 | 备注 |
| --- | --- | --- | --- | --- | --- | --- |
| 待补充 | 待补充 | 待补充 | 待补充 | 待补充 | 待补充 | 待补充 |
```

- 有多个 DP 时逐行展开。
- `功能点属性` 写枚举值、数值范围、单位等约束。
- `备注` 写互斥关系、联动逻辑、特例说明。

### 3.4 四、小程序功能说明

按页面分块，每个页面块由“说明段落 + 表格”组成。标题格式参考：

```md
### 1、（页面一）如：首页
```

页面下使用下表：

```md
| 序号 | 交互说明 | UI 图 |
| --- | --- | --- |
| 1 | 待补充交互说明 | 待补充示意图 |
| 2 | 待补充交互说明 | 待补充示意图 |
| 3 | 待补充交互说明 | 待补充示意图 |
```

- `交互说明` 写页面状态变化、按钮行为、DP 对应关系、异常提示等。
- `UI 图` 可以写截图说明、稿件链接或 `待补充示意图`。
- 默认至少保留 1 个页面块；用户明确给出多个页面时按顺序展开。

### 3.5 五、附件

```md
| 序号 | 附件 | 说明 |
| --- | --- | --- |
| 1 | 待补充附件 | 待补充说明 |
| 2 | 待补充附件 | 待补充说明 |
| 3 | 待补充附件 | 待补充说明 |
```

- 常见附件包括产品协议、DP 详细文档、竞品截图、品牌素材、评审纪要。
- 无附件时也保留表格和占位行。
