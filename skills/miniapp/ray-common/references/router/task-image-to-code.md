# 图片转代码路线卡

## 什么时候进入这里

- 用户提供截图、草图、设计稿、视觉参考图，希望生成 Ray 页面或组件代码
- 用户输入的主要信息源是“视觉结构”，而不是现成代码

## 执行顺序

1. 先做视觉拆解，不要立即写代码
2. 把页面拆成区块、控件、层级、状态
3. 判断哪些区域可优先使用业务组件，哪些可用 smart-ui，哪些需回退 Ray 原生标签
4. 再进入 UI 路线和规则卡查文档

## 推荐路径

1. 先读 `references/rules/tag-mapping.md`
2. 再读 `references/router/task-ui.md`
3. 优先在品类/业务 skill 中查找组件
4. 业务组件不满足时，查询 smart-ui skill
5. smart-ui 不满足时，回退到：`references/ray/INDEX.md` → `references/ray/component/<key>/` 或同名 `*.md`，再读目标组件文档

## 视觉拆解最少输出

- 页面级还是组件级
- 顶部区 / 内容区 / 底部区
- 列表、表单、卡片、按钮、弹层、空态等控件判断
- 明确哪些只是视觉推断，哪些已由文档确认

## 强约束

- 截图只能提供结构线索，不能证明某个 props、事件或 API 一定存在
- 若视觉上像某个组件，也必须查文档后才能使用
- 涉及文案时必须补读 `references/rules/i18n.md`

## 离开前检查

- 是否有把视觉猜测误写成文档事实
- 是否完成了语义到标签的校验
- 是否只读取了必要的组件文档
