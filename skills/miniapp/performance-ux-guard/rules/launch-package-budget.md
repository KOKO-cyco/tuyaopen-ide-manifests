# Package Budget

## Rule

主包体积 ≤2MB，单个分包 ≤2MB，总包体 ≤20MB；超出预算的资源必须外置到 CDN 或拆分。

**AI 不要自行估算包体**（本地 `dist/` 与线上实际包体差距可达 10-20×，原因见下方"关键陷阱"）。需要判断当前是否超预算时，按以下顺序向用户索取数据：

1. **优先获取真实数据**：让用户提供小程序发布平台的"UI 包体"页面链接（按 `miniProgramId` 查询），AI 读取其中主包、各分包、总包的实际上传大小。
2. **退而求其次**：让用户直接告知当前主包 / 分包 / 总包大小（用户从平台、`clix dev publish` 输出、或其他可信来源拿到），AI 据此判断是否超预算。
3. **完全无数据时**：明确告知用户"目前无法判断是否超预算"，给出**优化方向**（分包、CDN、清理、子 UI、按需加载等）和**优化后再次确认**的步骤，但不要给出具体"减少多少 MB"的结论。

## 关键陷阱：本地 `dist/` 不能作为包体参考

本地 `du -sh dist/` 的结果与线上真实上传包体**完全不可比**。实测案例：本地 `dist` 7.2 MB，线上实际 300+ KB（约 24× 差距）。原因：

- `*.map` source map 文件在 dev/preview 产物中保留，上传时剥离
- 公共依赖（React、Smart UI、Ray runtime 等）在容器层已沉淀，上传时由发布流程剔除（见 `launch-public-dependency-cache.md`、`launch-remove-react-dom.md`）
- 图片等大资源走 CDN，上传前会被排除（见 `launch-resource-cdn.md`）
- 上传前还有一道 minify / gzip / brotli 压缩
- dev 元数据（tsbuildinfo、构建 cache 等）在 dist 内但不上传

→ 因此**任何基于 `dist/` 大小的推断都是错的**。AI 见到 dist 体积请直接忽略，不要据此判断预算。

## Bad

```
开发同学："帮我看看包体超不超预算"
AI："看了下 dist 是 7.2MB，主包超了 2MB 预算 5.2MB..."   // ❌ 错误，dist ≠ 真实包体
```

或

```
AI 直接跑 `du -sh dist/` 输出 7.2MB，对照 2MB 预算给出"超 3.6×"的结论        // ❌ 错误，结论无效
```

## Good

```
开发同学："帮我看看包体超不超预算"
AI："本地 dist 与线上真实包体差距很大（source map / 公共依赖外置 / 大资源 CDN），无法据此判断。

请提供以下任一信息：
1. 小程序发布平台「UI 包体」页面链接（按 miniProgramId 查询）
2. 或直接告知当前主包 / 各分包 / 总包的大小（单位 MB / KB 均可）"

→ 用户提供："主包 1.6MB，分包 A 0.8MB，分包 B 1.2MB，总包 3.6MB"

AI："主包接近 2MB 预算上限，建议... 分包均健康，总包远低于 20MB 总预算。"
```

或在用户没有这些数据时：

```
AI："目前没法判断当前包体是否超预算。建议先做一次发布（哪怕是 dev/pre 环境）拿到平台 UI 包体页数据后再来评估。

不过即便不知道当前数值，仍有几个通用方向你可以先做："
- 检查是否分包：[…]
- 检查公共依赖是否走容器复用：[…]
- 检查重资源是否走 CDN：[…]
- 检查是否有 dead code：[…]
"
```

## Why

- 主包大小直接决定首屏下载耗时，每多 1MB 在 3G/弱网约多 1-2s 启动时间，预算是启动性能的硬约束。
- 但 AI 必须基于**真实包体数据**才能给有效建议；用 `dist/` 估算会得出系统性错误的结论，反而误导用户做无效优化（甚至破坏性"减包"操作）。
- "向用户索要数据"看似多一步交互，实际是避免 AI 凭空判断的关键护栏。
