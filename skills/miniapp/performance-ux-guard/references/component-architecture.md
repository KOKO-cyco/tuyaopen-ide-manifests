# Component Architecture（组件架构与拆分）

**实现大需求（涉及 2+ 页面、复杂交互、MasterGo 设计稿还原、或单页多分区复杂 UI）时，必须先做组件拆分再编码。**

**硬性要求：输出模块拆分和原子拆分后必须暂停，等待用户明确确认；确认前不得进入业务实现。**

## 拆分流程

```
大需求 ──→ 模块拆分 ──→ 确认 ──→ 原子组件拆分 ──→ 确认 ──→ 组装实现 ──→ 自检
```

### Round 1 — 模块拆分

1. 分析需求，识别独立功能模块
2. 提取可复用部分为公共组件候选
3. 输出模块划分方案，**询问开发者是否符合预期**
4. **停止继续编码，等待开发者确认**

输出格式：

```markdown
## 模块拆分方案
### 页面
- PageA: 描述
- PageB: 描述

### 公共组件候选
- ComponentX: 被 PageA/PageB 共用，可提取
- ComponentY: 通用交互模式，可复用

### 是否符合预期？
请确认后进入下一轮拆分。
```

### Round 2 — 原子组件拆分

1. 将每个模块进一步拆分为原子化组件
2. 定义每个组件的 Props 接口、状态管理方式、数据流
3. 标注哪些使用 Smart UI，哪些需自研
4. **再次询问开发者确认**
5. **停止继续编码，等待开发者确认**

### Round 3 — 组装实现

1. 按确认后的架构编码
2. 公共组件放在 `src/components/` 下
3. 页面级组件放在对应页面目录
4. 编码完成后执行 UI 自证；如果 `user-miniapp-devtools` MCP 可用，调用 MCP 收集快照、截图、日志和关键交互证据

## 拆分原则

- **单一职责**：每个组件只做一件事
- **可复用优先**：如果两个地方用到类似 UI/逻辑，提取为公共组件
- **Props 驱动**：组件行为通过 Props 控制，减少内部状态
- **组合优于继承**：通过组件组合（children/slot）实现灵活布局
- **Smart UI 优先**：通用 UI 用组件库，业务逻辑用自研组件包装
- **扩展成本评估**：轻量差异优先通过 Props/Slot/样式变量向下兼容扩展；大改会污染老组件或影响旧业务时，可以 fork / 新建组件并说明边界
